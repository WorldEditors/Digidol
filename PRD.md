1. Sample_User_Profile_Init函数： 采样 User Profile 至少包含如下维度，下面各个维度需要每个维度撰写一个采样函数，其中，具体分布特别是相对复杂的分布不应硬编码在采样函数中，而是用独立的数据文件，采样函数通过读入数据文件来进行采样。
* 出生年月日：按照30～40岁平均进行采样。<30岁的分布应随着年龄减小略微降低；>30岁的分布应逐年减小降低。
* 性别采样
* 受教育程度：随机采样。需要考虑年龄分布，如17岁高中毕业，不可能具有大学或博士学历。
* 专业： 包括文学 / 音乐 / 喜剧 / 历史 / 政治 / 地理 / 数学 / 物理 / 化学 / 生物 / 工程科学 （包括计算机、 人工智能、 软件工程、 生物工程、 制药、 等等），需要继续补充
* 身高体重：  应考虑合理的身高体重分布采样
* 当前住房/租房位置： 按照人口密度大致分布进行采样。中国和海外各自一半。需要下载一个人口密度分布的相关数据，读入该数据，并调用地图API来进行采样。支持人口密度数据随时更新
* 当前工作/学习单位和位置： 中国和海外各自一半，需要预先下载全球雇主数据，读入该数据然后进行采样。支持全球雇主数据持续更新
* 家庭成员： 父母 / 夫妻 / 儿女 是否健在， 是否异地， 包括家庭成员所在地点
* 财产： 包括不动产、车、衣物数量、一些设备等
* 平时兴趣/爱好分布，至少包含：
  ** 摄影： 没有摄影爱好/人像/风景等
  ** 游戏： 即时战略/角色扮演/对战/多人在线/日式/格斗等
  ** 电影： 包括剧情/动作/等等，请将该列表补充完整
  ** 动画/漫画： 包括日式/国产/欧美等等
  ** 运动： 包括无健身爱好/撸铁/游泳/街舞/中国舞/爵士舞/徒步/羽毛球/乒乓球/网球/篮球/足球/
  ** 听歌： 包括无/听歌/纯音乐/摇滚/蓝调/民谣等，继续补充
  ** 音乐爱好： 包括无/唱K/合唱/古筝/笛/唢呐/钢琴/小提琴/竖琴/大提琴/萨克斯/吉他/贝斯，继续补充。注意应该和听歌爱好相近而不是相冲突。注意会乐器的在人群占比较低
  ** 明星： 包括完全无感 / 影视明星 / 歌手 对于影视和歌手应下载目前海内外知名歌手列表，并按照人群分布制定采样策略
  ** 电视综艺： 包括看球赛 / 综艺节目 / 电视剧 / 不喜欢看电视。 下载当前国内海外较火的一些节目的名单，并读入名单和进行采样。注意各节目爱好的性别/年龄/学历可能存在的偏置问题。因此，名单里面应该包含性别年龄和学历的分布。
  ** 旅游： 包括出行方式的偏好 / 目的地偏好 （如山川自然风景，城市，小镇等），旅游的频度等。
  ** 手工： 如 无 / 陶艺 / 木艺等
  ** 购物： 线上购物/线下购物/经济适用基本不购物等等
* 宠物： 包括有养宠物（猫/狗/蜥蜴等）/ 不养但喜欢的宠物 / 讨厌的宠物等
* 喜好的食物，至少包含：
  ** 辣/不辣
  ** 重口/清淡
  ** 蒸/烤/炸
  ** 消费价位
* 衣品/穿着
  ** 运动 / 商务 / 休闲 / 隆重等
  ** 消费价位
* 工作日 / 学习日的日常生活计划，如常规在什么时间做什么事情。
* 一年中的大致安排， 包括常规工作日 / 寒暑假 / 非常规（如周末上班，工作日休息）， 待业在家， 退休等
* 长期病： 如健康 / 高血压 / 脂肪肝 / 肺病 / 皮肤病 / 癌症 / 自免疾病 / 糖尿病 / 痛风等
* 其他状态包括：
  ** 是否在一段恋情内
  ** 是否患病
  ** 是否失业
  ** 是否抑郁
  ** 是否有残疾

在上述生成函数内，应充分考虑生成函数的关联特性。
最后，存在一个总的调用函数，生成user刻画的json文件
上述逻辑，采用模版 + 数据文件结合的同时，必要时可以调用LLM辅助判断

2. Update_User_Profile函数： User Profile更新
对User Profile进行更新的函数。输入user的json文件，以及对应的event列表，来输出该event列表之后一段时间后的user profile

Profile的更新应该满足：a. 性别/喜好等不应变化 b. 部分字段会随着时间的变化而变化 c. 部分字段会随着event改变而改变。例如，搬家导致家庭住址变化，开始健身计划导致体重发生变化。

3. Generate_Event函数：  Event生成
Event会基于当前的User Profile采样随机一个或多个事件（事件数可以通过参数给定），这些事件大类应该包括但不限于
a. 生病
b. 升学 / 毕业
c. 就业
d. 工作变更
e. 购物
f. 参加某些活动，包括但不限于：约会 / 社区派对 / 团建 / 家庭聚餐 / 某些仪式 等等
g. 旅游
h. 出差
i. 探亲访友
j. 意外事故
k. 参加会议
l. 看演唱会
c. 某些关注点的状态改变，如某明星塌房。某感兴趣的动画片出新版本。

设计一种采样逻辑，需要保障所生成的event和user profile相对一致，例如，年龄和就业 / 毕业等相符。看演唱会可能和明星的偏好相符合。

在Event类型订了之后，每个event需要有详细属性信息。例如购物是线上 or 线下， 购买什么内容。出差去了什么地方。事件时间等

上述逻辑，采用一种尽量通用可扩展但又能保留足够多样性的方式实现。例如调用llm来实现。

4. Generate_Request函数： Request生成

给定用户当前的具体Event（列表），生成Request. Request类型包括但不限于：
a. 求推荐
b. 要求记录一些内容
c. 要求咨询或索取某些信息，包括天气，解读当前某些信息等等。
d. 要求联系或知会某些人
e. 要求安排某些活动的场地/时间等
f. 要求设置某些属性，包括闹钟、亮度、字体、提醒等等

上述逻辑，采用一种尽量通用可扩展但又能保留足够多样性的方式实现。例如调用llm来实现。

5. User_Respond函数：

给定User Profile, Request对应的历史以及未来一段时间内的Event列表， 以及当前会话Session的上下文， 用户给予Respond. 这些Respond包括但不限于：
a. 对系统的恢复进行引导
b. 对自身的需求进行澄清
c. 追加产生下一个Request
d. 如果助手给的结果不准确，进行引导、反馈或批评
e. 如果助手给的结果足够准确，且没有后续Request，退出或给一些正向反馈
f. 纯粹闲聊和调戏助手

上述逻辑，采用一种尽量通用可扩展但又能保留足够多样性的方式实现。例如调用llm来实现。
